1 # barelog
2 
3 **barelog** is a set of C99 modules that can be used to do some logging on many-core systems. The primary targets of barelog are the embedded heterogeneous many-core platforms (such as the [Parallella platform](https://parallella.org)) or any
4 core that is too small to run any Linux based OS, thus forbidding the use of traditional tools.
5 
6 The main use-case would be the logging of some calculus-specific cores that don't
7 have any kernel but **can still access a shared memory space** to interact with a
8 more "traditional" host (that is to say another CPU able to run a Linux kernel).
9 
10 Please note that due to it's current limitations, barelog is not meant to be used for serious, efficient logging/tracing. For a more sophisticated tool that provides very efficient tracing, please see also [barectf](https://github.com/efficios/barectf).
11 
12 Note : in the following document, the terms "host" and "target" refer respectively
13 to a system running a Linux kernel and able to initialize the shared memory and to
14 the specific core that doesn't run any kernel.
15 
16 **Key features**:
17 
18  * Entirely configurable: you have full control over the functions used by the
19  modules to interact with the shared memory as well as the total amount of
20  memory used by barelog (inside each core as well as in the shared section).
21  * Easy to use: a simple call of the **barelog_log()** function (after proper
22  initialization of the modules) allows you to log events without any further
23  complications.
24  * Provides several "functioning modes": you can enable/disable some parts of
25  the code to suit your needs. For example, to gain some performance, you might
26  want to disable the "DEBUG_MODE" that only offers some internal debugging
27  functions.
28  * Flush events whenever you want: a round-buffer allows you to store the events
29  in the local memory of the logged core as long as you want before actually
30  flushing them into the shared memory. You have full control over which stored
31  event to actually put into the shared space.
32  * Format the events data as you want: since the logging module use a modified
33  version of "snprintf" you can store any type of data (represented as a string)
34  in a event.
35 
36 **Current limitations**:
37 
38  * Pretty heavy impact on the performances: since the logging module use a
39  modified version of "snprintf", it's quite demanding in terms of clock cycles
40  to produce an event.
41  * The size of the actual event's data is statically fixed: that means that if
42  the events data are not full, there will be waste of both local memory
43  (of the logged core) and shared memory.
44  * The data of an event is represented by a string: which means that you can't
45  directly access to all the data logged into that event since they are wrapped
46  in a string.

48 ## Using
49 
50 ### Compiling the modules
51 
52  1. You first need to edit the **common/include/config.h** file to ensure that
53  barelog is configured to suit your needs. Note that you can directly include
54  a custom configuration header by placing it inside the **platforms**
55  directory and then including it.
56 
57  2. Once it's done, you may want need to edit the Makefile to properly set the
58  compiler used to compile the target module code. You can also set the
59  TARGET_CC flag during the 'make'.
60 
61  3. Then simply compile the modules using the provided Makefile. You can specify
62  whether or not to use a cross-compiling toolchain by setting the
63  CROSS_COMPILE flag. You can also decide if you rather want the resulting
64  libraries to be static (.a) or shared (.so) by setting the HLIBTYPE and/or
65  TLIBTYPE flags (where 'H' stands for Host and 'T' for Target). The default
66  behavior is to produce static libraries.
67 
68 
69  make
70 
71  Or
72 
73 
74  make HLIBTYPE=so TLIBTYP=a
75 
76 If everything went well, two libraries should have been produced in the **libs**
77 folder :
78 
79  * **libbarelog_host**: targets the host program.
80  * **libbarelog_logger**: targets the target program.
81 
82 ### Instrumenting and compiling your code
83 
84 #### Instrumenting your code
85 
86 Once you have compiled the modules, you just need to instrument your code to get started !
87 
88 To do that, you have to follow those steps :
89 
90  1. Initialize the host: you will have to create the **barelog_platform_t** along
91  with some memory management functions and to register them to the logger on
92  the host by calling the **barelog_host_init()** function. This will allocate
93  all the needed chunks of shared memory according to the "config" file and
94  initialize the all host module.
95 
96  2. Initialize the target: this basically involve the same steps as above but
97  with everything specific to the target.
98 
99  3. Instrument the target code: by using a combination of the **barelog_log()**
100  **barelog_flush()** and **barelog_clean()**, you should be able to produce and manage the events inside the logged core.
101 
102  4. Retrieve the events on the host: the host API offers some functions to
103  extract and display the logged events (please see the given example).
104 
105  5. Finalize the logger: once you're done logging around, use the
106  **barelog_finalize() function to ensure every resource is correctly
107  deallocated.
108 
109 Please refer to the documentation and/or the given example for more informations.
110 
111 **WARNING** : if you use barelog, some part of the shared memory (beginning at the
112 given platform's mem_space) will be used by it. To avoid every hazardous behavior,
113 consider using the **BARELOG_SHARED_MEM_MAX** macro (which give the size (in
114 bytes) of the memory taken by barelog) when allocating new chunks of memory for
115 your personal needs.
116 
117 #### Compiling your code
118 
119 Now that we have everything ready, we just need to compile our programs (one
120 running on the host and the other on the target).
121 
122 First of all, make sure that the previously generated barelog's libraries can be
123 found by the compiler/linker. Assuming that your using gcc, you just need to
124 specify the -L option :
125 
126  gcc -L path/to/libraries/
127 
128 You will then have to build the host program using the **libbarelog_host** library
129 and the target program with the **libbarelog_logger** library :
130 
131  gcc -L path/to/libraries/ target_main.c -lbarelog_logger
132  gcc -L path/to/libraries/ host_main.c -lbarelog_host
133 
134 Of course, this need to be adapted in case you need to use another compiler.
135 
136 ### Create your own configuration file
137 
138 To create you own configuration file, you can simply follow the "template" given
139 by **common/include/config.h**. You don't have to fulfill every fields since there
140 already is some default values (please refer to config.h).
141 
142 Once it's done, you just have to put it in the **platforms** directory, thus
143 guaranteeing that you could later include it in the config header.
144 
145 ### Warnings
146 
147  * The core numbering on the target must begin at 0.
148  * The barelog_device_mem_manager module should be placed in the local memory
149  of each logged core.
150  * The "SAFE" mode, providing shared memory synchronization mechanism is still
151  to be tested, thus implying that no guarantee can be provided.
152 
153 ## Configuring new behaviors/functionalities
154 
155 You might want to add some functionalities that need some data stored into the
156 shared memory space of barelog. Since this space is strictly ordered, you will
157 have to follow those steps to ensure the good global behavior of the modules :
158 
159  1. Define the size taken by those data inside the **barelog_internal.h** file:
160  you can use the following naming convention : 'BARELOG_FUNCNAME_MEM_SIZE'
161 
162  2. Edit the BARELOG_SHARED_MEM_DATA_OFFSET macro to take the new data in account
163  while computing the offsets of each barelog's data inside the shared memory.
164 
165  3. Reserve a new mem_space for your data by adding '1' to the
166  BARELOG_HOST_NB_MEM_SPACE macro.
167 
168  4. Define the new index inside the host's mem_space table of the new data:
169  you can use the following naming convention : 'BARELOG_FUNCNAME_I'. Please
170  be careful with the index since some may already have been taken and the
171  BARELOG_NB_CORES first refer to the actual events reserved memory spaces. You
172  can follow what has been done with BARELOG_DEBUG_MODE and BARELOG_SAFE_MODE
173  to get the global picture of how to do it.
174 
175  4. Modify the behavior of the "host_mem_manager_init()" and
176  "host_mem_manager_finalize()" functions to respectively init and finalize the
177  newly reserved mem_space.
